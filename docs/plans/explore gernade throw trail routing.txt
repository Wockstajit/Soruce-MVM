Swap Mapping:

CS2 Source: particles/entity/spectator_utility_trail.vpcf (the demo-playback utility trail anchor)
On/Less modes: nullptr → passes through vanilla CS2 (stock line, no swap)
Modern mode: swaps to particles/filmmaker/modern/arc9_fas_muzzleflashes/mvm_grenade_trail.vpcf

This is NOT a macro-based rule (no FXRULE / FXRULE_MODERN) — it's a hardcoded VariantRule struct because Modern-only swaps that pass through vanilla in On/Less modes don't fit the macro pattern.

5. MW2019 Grenade Trail Reference (from docs/mw2019-fx-mapping-reference.md)
Section 7 (Particle Source Files, line 212):

mw2019_effects.pcf | Misc COD-specific effects (grenade trails, molotov, etc.)
The source reference confirms MW2019 has grenade trails defined in mw2019_effects.pcf, but the actual PCF asset is not currently in the converted Modern pack — instead, the CS2 port synthesizes the trails dynamically via postprocess_modern._trail_system_text (lines 291–434), authoring from scratch with CP-speed gating (a CS2-specific refinement not found in the original GMod conversion).

Relevant Background from §6b (Barrel Rope Wisps, lines 144–174):
The reference documents that GMod's barrel_smoke_trail(_b) systems use C_OP_RenderRopes with ~5s lifetime, which causes rope-ribbon stretching artifacts in CS2. The Modern grenade trails intentionally use discrete sprites instead, matching the documented CS2-port gotcha mitigation strategy.
