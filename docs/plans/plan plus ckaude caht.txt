## Context

The Modern (MW2019/ARC9) particle pack has three loose ends the user wants closed, in
priority order of "make it look good" + "stop lagging my PC":

1. **Muzzle-flash burst looks dated.** The central "flash explosion" sprites still use
   `hl2_muzzleflash` (a 2004 Half-Life 2 asset, 7 frames) and never `PositionLock`, so the
   burst is low-res and drifts. The pack already ships a better flash sheet
   (`fas_muzzleflash_test_b`, 2048×1024, 18 frames) — some break/LMG leaves already use it.
   **User pick: Hybrid** — swap the remaining bursts to `fas_muzzleflash_test_b` AND adopt

2. **Thrown-grenade smoke trail.** Already exists (Modern-only, `spectator_utility_trail.vpcf`

   native CS2 voxel system and is **not touched by anything here** (confirmed) — it stays
   100% vanilla.
3. **Precache stall is back.** Swap targets resolve via a **synchronous blocking load, one

   **zero** new top-level swap targets, so they do not worsen precache — the recent growth was
   the 10 `_fp` files (kept). The throttle is the net fix.

## Part C — Precache stall fix (frame-time-aware drain)

**Where:** `AfxHookSource2/Filmmaker/Movie/ParticleFx.cpp`, `PumpMainThread()` drain block
(lines 220-231). This is the only correctness-adjacent change; keep it small and local.

**Problem:** the drain resolves exactly one target every frame with no regard to how long the
last one took or whether the game is under render pressure, so a run of heavy cold loads after
a demo load / mode switch produces a sustained multi-second, PC-wide stall.

**Design — replace the unconditional "1 per frame" with an adaptive, wall-clock-spread drain**
(all state main-thread-only statics; no new locks beyond the existing `g_mx` guard around the
queue pop):
- **While playing (`DemoIsPlaying() && !DemoIsPaused()`):** resolve **at most one target per
  `kResolveIntervalMs`** of wall time (e.g. 50ms → ~20/sec) instead of every frame, leaving
  recovery frames between heavy loads so FPS never floors.
- **Adaptive backoff:** time each `ResolveHandleOnMainThread` call; if it exceeds
  `kHeavyResolveMs` (e.g. 15ms), push the next allowed resolve out by a longer
  `kHeavyBackoffMs` (e.g. 150ms) so consecutive heavy assets can't stack into a freeze.
- **While paused (`DemoIsPaused()`):** the filmmaker's normal setup state has no live-render
  pressure — drain a **small time-budgeted burst** (loop resolving until ~6ms elapsed or the
  queue empties) so the queue clears quickly while the user is composing a shot, not during
  motion.
- Knobs as named `constexpr` at file scope so they're easy to tune; behavior is unchanged in
  correctness (only *when* resolves happen), so risk is low.

**Optional stretch (only if the trickle feels too slow in play):** priority-resolve the
currently spectated weapon's flash/tracer targets first by moving them to the front of
`g_resolveQueue`. Not required for the core fix; leave out unless verification shows a visible
first-shot vanilla flash.

**Do not** change the level-change full cache clear (ParticleFx.cpp:210-219) — it's required
because resources purge on map transition (CRASH LESSON 4). The throttle simply spreads the
requeue it produces.
